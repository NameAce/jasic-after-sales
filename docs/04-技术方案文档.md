## 二、项目架构

### 2.1 Maven 多模块结构

```
jasic-after-sales/               父工程（版本管理）
├── jasic-common/                通用模块（常量、枚举、异常、注解、工具类）
├── jasic-framework/             框架模块（Sa-Token、MyBatis-Plus、Redis、全局异常）
├── jasic-system/                系统管理模块（权限核心业务）
├── jasic-customer/              C端客户模块
├── jasic-admin/                 启动模块（Spring Boot入口 + 配置）
└── jasic-ui/                    前端工程（Vue2 + Element UI）
```

### 2.2 模块依赖关系

```
jasic-admin → jasic-system → jasic-framework → jasic-common
jasic-admin → jasic-customer → jasic-framework → jasic-common
```

### 2.3 各模块职责

- **jasic-common**：零业务依赖。常量（CacheConstants等）、枚举（SubjectTypeEnum/DataScopeEnum等）、自定义异常（ServiceException）、自定义注解（@DataScope/@OperLog）、基类（BaseEntity/Result/PageQuery）、工具类。不依赖Spring
- **jasic-framework**：技术中间件集成。Sa-Token配置与权限接口实现、MyBatis-Plus配置与数据权限插件、Redis配置、全局异常处理、操作日志AOP切面、错误码定义
- **jasic-system**：权限相关所有业务代码。认证登录、用户/角色/菜单/公司/签约/大区管理
- **jasic-customer**：C端客户业务。客户微信登录注册、客户信息管理
- **jasic-admin**：Spring Boot启动入口 + application.yml配置 + MyBatis XML映射文件

## 三、开发规范

### 3.1 工程分层

采用三层架构（不使用Manager层）：

| 层级 | 职责 | 示例 |
|------|------|------|
| Controller | 参数校验、请求转发，不放业务逻辑 | SysUserController |
| Service | 业务逻辑（含缓存处理、多DAO组合） | ISysUserService / SysUserServiceImpl |
| Mapper | 数据访问（MyBatis-Plus BaseMapper + 自定义XML） | SysUserMapper |

### 3.2 领域模型

| 类型 | 后缀 | 用途 | 示例 |
|------|------|------|------|
| DO | 无后缀或DO | 数据库表映射对象 | SysUser |
| DTO | DTO | Service间传输 | SysUserDTO |
| VO | VO | 返回前端展示 | SysUserVO |
| Query | Query | 查询参数封装 | SysUserQuery |

### 3.3 命名规范

**Service/Mapper 方法命名：**
- get：获取单个对象
- list：获取对象列表
- count：统计数量
- save：新增
- update：修改
- remove：删除

**包命名：**
- 基础包：`com.jasic.aftersales`
- 各模块：`com.jasic.aftersales.common`、`com.jasic.aftersales.framework`、`com.jasic.aftersales.system`、`com.jasic.aftersales.customer`

### 3.4 注释规范

所有类必须添加 Javadoc 类注释：
```java
/**
 * 用户管理 Service
 *
 * @author Zoro
 * @date 2026/03/18
 */
```

所有方法必须添加 Javadoc 方法注释：
```java
/**
 * 根据用户ID查询用户信息
 *
 * @param userId 用户ID
 * @return 用户信息VO
 */
```

所有枚举字段必须有注释说明用途。

### 3.5 接口规范

**统一返回格式：**
```json
{
  "code": "00000",
  "msg": "操作成功",
  "data": {}
}
```

**错误码规范（5位）：**
- `00000`：成功
- `A0001`~`A9999`：用户端错误（如参数校验失败、未登录、无权限）
- `B0001`~`B9999`：系统内部错误
- `C0001`~`C9999`：第三方服务错误

**分页请求：**
```json
{
  "pageNum": 1,
  "pageSize": 10,
  "orderByColumn": "create_time",
  "isAsc": "desc"
}
```

**分页响应：**
```json
{
  "code": "00000",
  "msg": "操作成功",
  "data": {
    "records": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 10
  }
}
```

## 四、核心技术实现

### 4.1 Sa-Token 多账号体系

B端和C端使用 Sa-Token 的多账号体系功能：

```
B端：StpUtil（默认体系）
  - Token前缀：satoken
  - 登录：StpUtil.login(userId)
  - Session存储：currentCompanyId、currentTypeCode、currentSubjectType

C端：StpCustomerUtil（自定义体系）
  - Token前缀：customer
  - 登录：StpCustomerUtil.login(cUserId)
  - 无RBAC权限
```

### 4.2 权限接口实现

实现 Sa-Token 的 `StpInterface`，从 Redis 加载用户权限：

```java
@Override
public List<String> getPermissionList(Object loginId, String loginType) {
    Long userId = Long.valueOf(loginId.toString());
    Long companyId = SecurityContext.getCurrentCompanyId();
    // 从 Redis 获取 user:perms:{userId}:{companyId}
    // 若缓存不存在则从DB加载并缓存
    return permissionService.listPermsByUserAndCompany(userId, companyId);
}
```

### 4.3 数据权限实现

使用 MyBatis-Plus 的 `DataPermissionInterceptor`：

1. 自定义 `@DataScope` 注解标记需要数据过滤的Mapper方法
2. `DataScopeAspect` 切面解析注解，将过滤条件写入 ThreadLocal
3. `DataScopeHandler` 实现 `DataPermissionHandler` 接口，从 ThreadLocal 读取条件，动态拼接 SQL WHERE

```java
@DataScope(companyAlias = "t", userAlias = "t")
List<WorkOrderDO> selectWorkOrderList(@Param("query") WorkOrderQuery query);
```

### 4.4 Redis 缓存设计

| Key 格式 | 用途 | 过期时间 |
|----------|------|---------|
| `user:perms:{userId}:{companyId}` | 用户权限集合 | 登出/切换时删除 |
| `user:menus:{userId}:{companyId}` | 用户菜单树 | 登出/切换时删除 |
| `user:info:{userId}` | 用户基本信息 | 30分钟 |
| `company:children:{companyId}` | 公司下级公司ID列表 | 30分钟 |
| `region:companies:{regionId}` | 大区下网点ID列表 | 30分钟 |

### 4.5 全局异常处理

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    // Sa-Token NotLoginException → A0100 未登录
    // Sa-Token NotPermissionException → A0200 无权限
    // ServiceException → 自定义业务异常
    // MethodArgumentNotValidException → A0400 参数校验失败
    // Exception → B0001 系统内部错误
}
```

### 4.6 操作日志实现

通过 `@OperLog` 注解 + AOP 切面自动记录：

```java
@OperLog(title = "用户管理", operType = OperType.INSERT)
@SaCheckPermission("system:user:add")
@PostMapping
public Result<Void> save(@RequestBody SysUserDTO dto) {
    // ...
}
```

切面在方法执行前后记录请求参数、响应结果、耗时、操作人等信息，异步写入 `sys_oper_log`。

## 五、认证流程

### 5.1 B端PC登录

```
1. POST /auth/login {username, password}
2. 校验用户名密码
3. 查询 sys_user_company 获取关联公司列表
4. 若仅1个公司 → StpUtil.login(userId)，写入 Session
5. 若多个公司 → 返回公司列表，等待选择
6. POST /auth/choose-company {companyId}
7. 加载该公司下的角色和权限 → 缓存到 Redis
8. 返回 Token + 用户信息 + 菜单树
```

### 5.2 B端小程序登录

```
1. 小程序端 wx.login() 获取 code
2. POST /auth/mp-login {code}
3. 后端调用微信接口换取 openid
4. 通过 openid 查询 sys_user
5. 若匹配到 → 走上述流程（多公司选择等）
6. 若未匹配 → 返回"未绑定"，引导绑定
```

### 5.3 C端小程序登录

```
1. 小程序端 wx.login() 获取 code
2. POST /customer/auth/login {code, phoneCode}
3. 后端换取 openid + 手机号
4. 通过 openid 查询 c_user
5. 若存在 → StpCustomerUtil.login(cUserId)
6. 若不存在 → 自动创建 c_user → 登录
7. 返回 Token
```

### 5.4 切换公司

```
1. POST /auth/switch-company {companyId}
2. 校验用户是否关联该公司
3. 清除旧公司的权限缓存
4. 更新 Session 中的 currentCompanyId
5. 加载新公司的角色和权限 → 缓存到 Redis
6. 返回新的菜单树
```

## 六、前端技术方案

### 6.1 基础框架

基于 Vue2 + Element UI，参考若依前端架构：

- **构建工具**：Vue CLI
- **状态管理**：Vuex
- **路由**：Vue Router（动态路由）
- **HTTP**：Axios（封装拦截器）

### 6.2 动态路由

```
1. 登录成功 → 获取菜单树（后端返回）
2. 将菜单树转换为 Vue Router 路由配置
3. 动态注册路由：router.addRoutes(dynamicRoutes)
4. 菜单组件渲染侧边栏
```

### 6.3 权限控制

**按钮级权限指令：**
```html
<el-button v-hasPerms="['system:user:add']">新增</el-button>
```

**路由级权限：**
- 路由守卫中判断 Token 是否存在
- 无 Token → 跳转登录页
- 有 Token → 判断是否已加载菜单
- 未加载 → 请求菜单接口 → 动态注册路由

### 6.4 切换公司组件

顶部导航栏右侧下拉组件：
- 显示当前公司名称
- 下拉展示关联公司列表
- 点击切换 → 调用切换公司接口 → 刷新页面重新加载菜单和权限

## 七、部署方案

### 7.1 开发环境

- 后端：`java -jar jasic-admin.jar --spring.profiles.active=dev`
- 前端：`npm run serve`（开发模式，代理后端接口）
- MySQL + Redis：本地或Docker

### 7.2 生产环境

- 后端：JAR包部署，Nginx反向代理
- 前端：`npm run build` 产出静态文件，Nginx托管
- MySQL + Redis：独立服务器

### 7.3 配置文件分离

- `application.yml`：公共配置
- `application-dev.yml`：开发环境（本地MySQL/Redis地址）
- `application-prod.yml`：生产环境（生产MySQL/Redis地址）
