# 佳士售后系统 - 权限架构设计文档

> 本文档详细描述系统的权限架构设计，包括账号体系、组织模型、RBAC权限模型、数据权限、菜单设计等核心内容。

## 二、账号体系设计

### 2.1 B端/C端分离

系统采用 B端（内部员工）与 C端（终端客户）账号完全分离的策略：

| 维度 | B端（sys_user） | C端（c_user） |
|------|-----------------|--------------|
| 登录方式 | 用户名/密码（PC）、微信openid（小程序） | 微信openid/unionid |
| 使用端 | PC后台、企业小程序 | 客户小程序 |
| 权限体系 | 完整RBAC，按钮级 | 无（仅看自己的工单） |
| 数据关联 | 公司/角色/大区 | 工单/设备 |
| Token体系 | Sa-Token 默认体系（StpUtil） | Sa-Token 多账号体系（StpCustomerUtil） |

**分离理由：**
1. 认证方式天然不同，强行统一会使RBAC变复杂
2. C端不需要角色权限管理
3. 安全隔离更干净

### 2.2 B端多端登录

B端员工可通过两种方式登录：

- **PC端**：用户名 + 密码登录
- **企业小程序**：微信授权 → 通过 openid 匹配 sys_user → 完成登录

两端共享同一套RBAC权限体系，权限标识（perms）全端通用。PC端通过动态菜单控制页面级权限，小程序端通过返回 perms 集合由前端静态控制按钮显隐。

### 2.3 C端登录

客户小程序：微信授权 → 获取手机号 → 自动创建/匹配 c_user → 登录。

## 三、组织模型设计

### 3.1 公司分类体系

引入 **type_code**（细粒度分类）和 **subject_type**（粗粒度分组）双层分类：

```
subject_type（主体类型）        type_code（公司类型编码）
    │
    ├── PLATFORM ──────────── PLATFORM（平台）
    │
    ├── HQ ────────────────── HQ_A（总部A）
    │                         HQ_B（总部B）
    │                         HQ_C（总部C）
    │                         HQ_D（总部D）
    │
    └── SERVICE ───────────── FIRST（一级服务网点）
                              SECOND（二级服务网点）
```

- **subject_type** 决定：菜单全集上限、数据权限SQL注入逻辑（HQ看签约网点数据、SERVICE看自身数据）
- **type_code** 决定：在主体类型菜单上限内的二次裁剪（通过 `sys_type_code_menu` 表）、使用哪套角色模板

### 3.2 公司关系

```
总部A ──签约──┬── 一级网点1 ──从属──┬── 二级网点1
              │                    └── 二级网点2
              └── 一级网点2 ──从属──── 二级网点3

总部B ──签约──┬── 一级网点1（同一个一级可签约多个总部）
              └── 一级网点3
```

- 总部与一级：多对多签约（`hq_first_contract` 表，含 region_id 大区绑定）
- 一级与二级：一对多从属（`first_second_relation` 表）

### 3.3 大区模型

- 大区归属于某个总部公司（`sys_region.company_id`）
- 一级网点与总部签约时绑定大区（`hq_first_contract.region_id`）
- 只有一级有大区概念，二级没有（但二级数据跟随一级的大区归属）
- 区域经理通过 `sys_user_region` 关联大区，只能看到负责大区下的数据

## 四、RBAC 权限模型

### 4.1 核心实体关系

```
用户(sys_user) ──N:N── 公司(sys_company)           通过 sys_user_company
用户(sys_user) ──N:N── 角色(sys_role)             通过 sys_user_role
角色(sys_role) ──N:N── 菜单(sys_menu)             通过 sys_role_menu
角色(sys_role) ──N:1── 公司(sys_company)           角色归属公司
菜单(sys_menu) ──N:1── 主体类型(subject_type)      菜单按主体类型划分
公司类型(type_code) ──N:N── 菜单(sys_menu)        通过 sys_type_code_menu（菜单上限）
```

### 4.2 角色设计

每个公司拥有独立的角色集合：

- **系统角色**：由平台通过角色模板自动创建，不可删除，可修改菜单分配
- **自定义角色**：由公司管理员自行创建，菜单从本 type_code 的菜单上限池（sys_type_code_menu）中勾选
- **角色字段**：
  - `company_id`：归属公司
  - `role_key`：角色标识
  - `data_scope`：数据范围（ALL/REGION/SELF）
  - `is_system`：是否系统角色

### 4.3 角色模板

平台管理员按 type_code 预设角色模板：

- 创建公司时，根据其 type_code 找到对应模板，自动在公司下创建角色并分配菜单
- 模板更新同步策略：
  - 模板**删除**菜单 → 所有该 type_code 公司的角色中同步移除该菜单
  - 模板**新增**菜单 → 不自动添加到公司角色，由公司管理员自行分配
  - 受影响的在线用户强制下线

### 4.4 菜单设计

```
菜单表 sys_menu 结构：
├── subject_type  所属主体类型（PLATFORM/HQ/SERVICE，仅3种）
├── menu_type     类型：M(目录) / C(菜单) / F(按钮)
├── parent_id     上级菜单ID
├── menu_name     菜单名称
├── path          路由地址（M/C级）
├── component     组件路径（C级）
├── perms         权限标识（F级，如 system:user:list）
├── icon          图标（M/C级）
├── order_num     排序
└── is_visible    是否可见

公司类型-菜单上限表 sys_type_code_menu 结构：
├── type_code     公司类型编码（HQ_A/HQ_B/.../FIRST/SECOND）
└── menu_id       菜单ID（从对应 subject_type 的菜单全集中勾选）
```

关键设计决策：
- **不使用 app_type**：权限标识 perms 全端通用（PC和小程序共用）
- **按 subject_type 定义菜单全集**：仅需维护 3 套菜单树（PLATFORM/HQ/SERVICE）
- **按 type_code 二次裁剪**：通过 `sys_type_code_menu` 表在主体类型菜单上限内分配每个公司类型的可用菜单
- **三级结构**：目录 → 菜单 → 按钮（与若依风格一致）

权限传递链路：
```
菜单全集（按 subject_type 定义）
    ↓ 裁剪
type_code 菜单上限（sys_type_code_menu）
    ↓ 裁剪
角色模板（sys_role_template_menu，基于 type_code）
    ↓ 裁剪
具体角色（sys_role_menu）
    ↓ 分配
用户权限
```

### 4.5 权限校验机制

**操作权限（接口级）：**
- 使用 `@SaCheckPermission("system:user:list")` 注解
- Sa-Token 从 Redis 获取当前用户在当前公司下的 perms 集合
- 判断是否包含所需权限标识

**按钮权限（前端）：**
- PC端：自定义指令 `v-hasPerms="['system:user:add']"`
- 小程序端：后端返回 perms 集合，前端判断显隐

## 五、数据权限设计

### 5.1 data_scope 机制

每个角色有 `data_scope` 字段，控制该角色能看到的数据范围：

| data_scope | 含义 | 适用场景 |
|-----------|------|---------|
| ALL | 全部数据 | 公司管理员 |
| REGION | 大区数据 | 总部区域经理 |
| SELF | 仅自己数据 | 维修员 |

### 5.2 多角色冲突处理

当用户在同一公司有多个角色且 data_scope 不同时，取**最小范围**：

```
SELF < REGION < ALL
```

即如果同时有 ALL 角色和 REGION 角色，生效的是 REGION。

### 5.3 SQL 注入逻辑

通过 MyBatis-Plus 的 `DataPermissionInterceptor` 自动注入 WHERE 条件：

**总部（HQ）场景：**
- `ALL`：`WHERE hq_company_id = {当前公司ID}`
- `REGION`：`WHERE hq_company_id = {当前公司ID} AND company_id IN ({大区下的一级ID及其二级ID})`
- `SELF`：`WHERE hq_company_id = {当前公司ID} AND assigned_user_id = {当前用户ID}`

**一级网点（SERVICE_FIRST）场景：**
- `WHERE company_id IN ({自己的ID} ∪ {下属二级ID})`

**二级网点（SERVICE_SECOND）场景：**
- `WHERE company_id = {自己的ID} OR origin_company_id = {自己的ID}`

### 5.4 大区数据范围

区域经理的数据范围推导：
```
用户 → sys_user_region → 关联大区ID列表
大区ID → hq_first_contract → 该大区下的一级网点ID列表
一级网点ID → first_second_relation → 下属二级网点ID列表
最终 WHERE：company_id IN (所有一级ID + 所有二级ID)
```

## 六、一人多公司设计

### 6.1 数据模型

- `sys_user` 表不存储 `company_id`
- 通过 `sys_user_company` 中间表建立用户-公司多对多关系
- 每条关联记录有 `is_default` 标识默认公司

### 6.2 登录与切换流程

```
1. 用户登录 → 验证凭证
2. 查询 sys_user_company → 获取关联公司列表
3. 若只有1个公司 → 自动进入
4. 若有多个公司 → 前端展示列表，用户选择
5. 选择后 → 将 currentCompanyId 写入 Sa-Token Session
6. 加载该公司下的角色和权限 → 缓存到 Redis
7. 切换公司 → 清除旧缓存，重新加载新公司权限
```

### 6.3 缓存隔离

Redis 缓存 key 按公司维度隔离：
- `user:perms:{userId}:{companyId}` — 用户在某公司下的权限集合
- `user:menus:{userId}:{companyId}` — 用户在某公司下的菜单树

切换公司时：清除旧 key，生成新 key。

## 七、权限变更与强制下线

### 7.1 触发强制下线的场景

1. 管理员修改角色的菜单分配
2. 管理员修改用户的角色分配
3. 平台模板同步移除菜单
4. 平台调整某 type_code 的菜单上限

### 7.2 处理流程

```
权限变更 → 计算受影响用户ID列表 → 调用 Sa-Token 踢人接口 → 清除 Redis 缓存
前端 Token 失效 → 弹窗提示"权限已变更，请重新登录"
```

## 八、SecurityContext 设计

封装全局上下文工具类，统一获取当前用户信息：

```java
SecurityContext.getCurrentUserId()          // 当前用户ID
SecurityContext.getCurrentCompanyId()       // 当前操作公司ID
SecurityContext.getCurrentSubjectType()     // 当前主体类型（HQ/SERVICE/PLATFORM）
SecurityContext.getCurrentTypeCode()        // 当前公司类型编码
SecurityContext.getCurrentPerms()           // 当前权限集合
SecurityContext.getCurrentRegionIds()       // 当前用户负责的大区ID列表
```

数据来源：Sa-Token Session，登录/切换公司时写入。
