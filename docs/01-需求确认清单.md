# 佳士售后系统 - 需求确认清单

> 本文档汇总了前期讨论中所有已确认的需求决策点，作为后续设计和开发的基线依据。

---

## 一、业务背景

| 项目 | 说明 |
|------|------|
| 公司性质 | 传统制造业，ToB，销售电焊机 |
| 系统目标 | 搭建售后服务系统，覆盖终端用户申请售后、服务网点维修、总部监控全流程 |
| 使用端 | PC管理后台（B端）、企业小程序（B端维修员）、客户小程序（C端终端用户） |

---

## 二、组织架构

### 2.1 公司层级

- **总部公司**：目前有4个（A、B、C、D），平级关系，负责不同业务但有交集
- **一级服务网点**：N个，与总部为多对多签约关系
- **二级服务网点**：N个，与一级为一对多从属关系

### 2.2 数据隔离规则

- [x] 四个总部公司数据完全隔离
- [x] 一级的N个公司数据完全隔离
- [x] 二级的N个公司数据完全隔离
- [x] 总部可以看到签约的一级公司数据
- [x] 一级可以看到下属二级公司数据
- [x] 数据隔离使用 `company_id` 字段区分，不使用 `tenant_id`

### 2.3 总部扩展性

- [x] 未来3-5年总部数量不会变化，但系统支持新增总部
- [x] 总部由平台管理员在系统中自助创建

---

## 三、核心业务流程

### 3.1 标准工单流程

1. 终端用户在小程序扫机器条码申请售后
2. 根据定位选择附近的服务网点（一二级都可选，暂不限制签约体系）
3. 服务网点接单并指派给内部维修员
4. 维修员维修完成，关闭工单
5. 终端用户查看后评价
6. 总部全程可见（只要是总部的设备，不管哪个网点在修）

### 3.2 升单（转移）流程

- 二级无法维修 → 升单给所属一级（工单转移，二级变为只读）
- 一级无法维修 → 升单给签约总部（工单转移，一级和二级都变为只读）
- 所有条码可解析到具体归属总部，升单时只能选择该条码归属的总部
- 升单本质是"转移"工单，转移后原网点只读

### 3.3 总部操作权限

- [x] 非升单到自己的工单 → 总部只能看，不能操作
- [x] 升单到自己的工单 → 总部可以操作

---

## 四、用户与账号体系

### 4.1 B端/C端分离

- [x] B端（内部员工）和 C端（终端客户）账号完全分离
- [x] B端使用 `sys_user` 表，C端使用 `c_user` 表
- [x] 分离理由：认证方式不同、权限体系不同、安全隔离更干净

### 4.2 B端账号

| 项目 | 说明 |
|------|------|
| 登录方式 | PC端：用户名/密码；小程序端：微信授权（openid 绑定 sys_user） |
| 使用端 | PC后台 + 企业小程序 |
| 权限体系 | 完整RBAC，按钮级权限，包括小程序端 |
| 数据关联 | 公司/角色/大区 |

### 4.3 C端账号

| 项目 | 说明 |
|------|------|
| 登录方式 | 微信授权自动登录，获取手机号自动创建用户 |
| 使用端 | 客户小程序 |
| 权限体系 | 无（或极简），只能看自己的工单 |
| 数据关联 | 工单/设备 |

### 4.4 一人多公司

- [x] 总部场景存在一人负责多个总部业务
- [x] 同一员工在不同公司可以有不同角色
- [x] 登录后选择当前操作公司，支持切换

## 五、权限设计确认

### 5.1 四个主体

| 主体 | type_code | subject_type | 说明 |
|------|-----------|-------------|------|
| 平台 | PLATFORM | PLATFORM | 超级管理员，维护基础数据和账号 |
| 总部A | HQ_A | HQ | 四个总部菜单各自独立 |
| 总部B | HQ_B | HQ | |
| 总部C | HQ_C | HQ | |
| 总部D | HQ_D | HQ | |
| 一级网点 | FIRST | SERVICE | 所有一级网点菜单统一 |
| 二级网点 | SECOND | SERVICE | 所有二级网点菜单统一 |

### 5.2 角色权限

- [x] 每个公司有自己内部的RBAC
- [x] 由平台管理员按 type_code 预设角色模板，创建公司时自动分配
- [x] 公司可自定义角色并分配菜单，但分配上限为该公司 type_code 的菜单上限（从主体类型菜单全集中裁剪）
- [x] 需要做到按钮级权限粒度
- [x] 菜单按主体类型（PLATFORM/HQ/SERVICE）维护，仅3套菜单树
- [x] 各公司类型（HQ_A/HQ_B等）通过 sys_type_code_menu 表在主体类型菜单上限内做二次裁剪

### 5.3 菜单设计

- [x] 菜单表包含三级：目录(M)、菜单(C)、按钮(F)
- [x] 按钮级包含 `perms` 权限标识
- [x] 小程序端不使用动态菜单，后端返回 perms 集合，前端静态控制
- [x] 权限标识 perms 全端通用（PC和小程序共用一套权限标识）
- [x] 不需要 app_type 区分，类型都是 ALL

### 5.4 数据权限

- [x] 角色包含 `data_scope` 字段：ALL（全部）、REGION（大区）、SELF（仅自己）
- [x] 多角色冲突时取最小范围：SELF < REGION < ALL
- [x] 总部区域经理只能看到负责大区下的数据（含一级下属的二级数据）

### 5.5 大区

- [x] 只有一级网点有大区概念，二级没有
- [x] 一级网点与总部签约时说明属于哪个大区
- [x] 区域经理只能看到该大区下的客户数据（含二级）

### 5.6 角色模板同步

- [x] 角色模板更新后需同步到已有公司
- [x] 平台移除某个权限时，该类型下所有公司角色中该菜单同步移除
- [x] 被移除权限的在线用户强制下线

### 5.7 权限变更处理

- [x] 权限变更时"强制下线"，提示用户"权限已变更，请重新登录"

### 5.8 多公司切换

- [x] 登录后如果关联多个公司，选择当前操作公司
- [x] 权限缓存按公司隔离：`user:perms:{userId}:{companyId}`
- [x] 切换公司时注意并发问题

## 八、开发规范确认

- [x] 工程分层三层：Controller → Service → Mapper/DAO（不使用Manager层）
- [x] 领域模型命名：DO/DTO/VO/Query
- [x] 方法命名：get/list/count/save/update/remove
- [x] 注释规范：所有类和方法使用Javadoc，@author Zoro，@date 为写入时日期
- [x] 数据库规范：遵循阿里手册（表必备三字段、索引命名、不使用外键等）
- [x] 不使用 `select *`
- [x] Long类型返回前端使用String（防精度丢失）
